# sc-holotable -- benchmarks

> Compile-time and runtime benchmarks for `sc-extract` +
> `sc-extract-generated`. Measurements are **run manually** via
> [`tools/bench/bench.ps1`](../tools/bench/bench.ps1); see its
> [README](../tools/bench/README.md) for usage.
>
> The **Latest results** section further down is auto-generated by the

## Running the benchmarks

```powershell
# From the workspace root. Full suite, ~30-60 minutes.
# -KillRa stops rust-analyzer before measuring (required for clean numbers).
.\tools\bench\bench.ps1 -Mode all -KillRa
```

The script writes:

- **`target/bench-results.json`** -- structured raw data, full environment snapshot, per-run timestamp.
- **`target/bench-history/<timestamp>.json`** -- timestamped copy (append-only history, never overwritten).
- **This file's *Latest results* section** -- auto-replaced in place via markdown markers.

When you run the script, the previous run's "Latest results" tables
are **automatically archived** into the History section below, then
overwritten with the new numbers. Pass `-NoAutoArchive` to suppress.

## What the modes measure

| Mode | Measures | Cost |
|---|---|---|
| `check` | Cold `cargo check -p sc-extract` for each feature set | ~10 min |
| `build` | Cold `cargo build -p sc-bench --profile X` for each profile x feature set | ~30 min |
| `runtime` | `sc-bench` against real `Data.p4k`, with + without reference graph | ~5 min |
| `all` | Everything above, chained | ~45 min |

## Memory sampling

The script samples `WorkingSet64` across all `cargo`/`rustc`/`link`/`lld-link`/`sc-bench`
processes every 500 ms and reports the peak total. This reflects the
true system-wide RAM high-water mark for the build.

**Caveats:** 500 ms granularity (sub-500 ms peaks during fast link
phases sometimes missed); under `-IgnoreRa` the sample includes
whatever rust-analyzer is running concurrently.

## Release profiles (established 2026-04-16)

Two profiles, tuned via per-dimension sweeps:

| Profile | LTO | panic | gen opt | Use case |
|---|---|---|---|---|
| `release` | thin | unwind | 1 | Day-to-day builds. Best compile/runtime compromise. |
| `release-max` | fat | abort | 1 | Shipped binaries / CI snapshots. No backtraces. |

`sc-extract-generated` gets `opt-level = 1` in both profiles because
fat LTO does the real optimization at link time, making the per-crate
opt level on generated code irrelevant to runtime while `opt=1` saves
~9% compile time vs `opt=3`.

### Key findings from the sweep (2026-04-16)

- **Fat vs thin LTO:** essentially tied on compile time even at `full`
  (+2.2% for fat). Fat marginally faster at runtime (-6% parse).
  Post-derive-drop IR volume is small enough per type that fat LTO's
  single-threaded link pass doesn't dominate.
- **`sc-extract-generated` opt=1 vs opt=3:** indistinguishable on both
  `entities-scitem-ships` and `full`. Fat LTO at link time makes
  per-crate opt level irrelevant. Only opt=0 is a useful lever
  (~34% faster compile, ~10% slower runtime).
- **`sc-extract` opt=1:** clear runtime regression (+50%). Hand-written
  algorithms (graph BFS, tag tree, locale parse) need full LLVM
  optimization that LTO can't retroactively apply.
- **Derive-drop** (commit `06e0d2e`): the biggest single win. Cold full
  check 4m 19s -> 37.66s (-85%); peak RAM 11.7 GB -> 2.54 GB.
- **Snapshot v5 byte-bundle:** eliminated `rmp_serde` monomorphization.
  Full cold build 3h 26m -> 7m 10s; peak RAM 25.7 GB -> 13.6 GB.
  (Then derive-drop brought it further to ~2m 54s / 4.4 GB.)

## Interpretation

- **Default cold check should be significantly faster than full.**
  With no features enabled, rustc strips most of `multi_feature/`
  during parsing. If default approaches full, the classifier has
  drifted toward schema-static.
- **`dormant` should be marginally slower than `full`.** Adds 809
  types whose references are already in-scope via the `full` forward.
- **Runtime parse under default returns 0 records.** Not a bug --
  no record extractor dispatch is compiled without leaf features.
- **Runtime parse under `--features full` is the realistic consumer
  number.** This is what a snapshot-building CI job or sc-weapons sees.
- **Snapshot size with graph >> without.** The reference graph
  (~1.4M edges) dominates the msgpack payload.
