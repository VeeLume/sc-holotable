# sc-holotable — benchmarks

> Compile-time and runtime benchmarks for `sc-extract` +
> `sc-extract-generated`. Measurements are **run manually** via
> [`tools/bench/bench.ps1`](../tools/bench/bench.ps1); see its
> [README](../tools/bench/README.md) for usage.
>
> The **Latest results** section further down is auto-generated by the
> script — don't hand-edit it. Everything else on this page (methodology,
> history, interpretation) is hand-written and preserved across runs.

## Running the benchmarks

```powershell
# From the workspace root. Full suite, ~30–60 minutes.
# -KillRa stops rust-analyzer before measuring (required for clean numbers).
.\tools\bench\bench.ps1 -Mode all -KillRa
```

The script writes:

- **`target/bench-results.json`** — structured raw data, full environment snapshot, per-run timestamp.
- **`target/bench-history/<timestamp>.json`** — timestamped copy (append-only history, never overwritten).
- **This file's *Latest results* section** — auto-replaced in place via markdown markers.

When you run the script, the previous run's "Latest results" tables
are **automatically archived** into the History section below, then
overwritten with the new numbers. No manual copy step needed. Pass
`-NoAutoArchive` to suppress the auto-archive.

## What the modes measure

| Mode | Measures | Cost |
|---|---|---|
| `check` | Cold `cargo check -p sc-extract` for each feature set | ~10 min |
| `build` | Cold `cargo build -p sc-bench --profile X` for each profile x feature set | ~30 min |
| `runtime` | `sc-bench` against real `Data.p4k`, with + without reference graph | ~5 min |
| `all` | Everything above, chained (build -> runtime per profile x feature) | ~45 min |

See [`tools/bench/README.md`](../tools/bench/README.md) for the full
definition of each mode (what gets cleaned, what gets timed, how feature
sets are routed).

## Memory sampling

The script samples `WorkingSet64` across all `cargo`/`rustc`/`link`/`lld-link`/`sc-bench`
processes every 500 ms and reports the peak total. This is a sum across
concurrent processes (rustc often runs multiple codegen units in
parallel) so it reflects the true system-wide RAM high-water mark for
the build.

**Caveats to keep in mind when reading Peak RAM:**

- Sampling has a 500 ms granularity. Sub-500 ms peaks during fast link
  phases are sometimes missed. Long-running rustc phases
  (monomorphization, typecheck) plateau for many seconds, so in
  practice this matters only at the very end of a build.
- The script samples processes by name, not by parent-child tree. Under
  `-KillRa` this is accurate because rust-analyzer has been stopped.
  Under `-IgnoreRa` the sample includes whatever rust-analyzer is
  running concurrently, which inflates the reported peak. That's
  intentional — it matches the "day-to-day with IDE" reading.
- Working set is committed RSS, not virtual size. It's the number that
  matches Task Manager's memory column and what matters for OOM risk.

## Findings from 2026-04-15

Three things surfaced from the first clean (RA-paused) benchmark run
that are worth investigating before starting on `sc-weapons`:

### Full dev-opt build regression

Pre-polymorphism (v1 generator, 1,935 types): **20m 56s** cold.
Post-polymorphism (v2 generator, 6,234 types): **3h 25m 58s** cold.

That's a ~10× regression. `cargo check --features full` at 4m 5s
scales in line with the 3.2× type-count increase, so the typecheck
path is roughly proportional. The `dev-opt` build path is not — which
points at codegen, not typecheck, as the dominant cost. Likely
suspects: serde-derive monomorphization of 6k structs + ~1.2k enums,
or `sc-extract-generated`'s `opt-level = 1 + codegen-units = 256`
interacting badly with the new type count (256 codegen units × many
small units might serialize poorly on an 8-core box).

Action items worth trying, rough cost/benefit:

- **`cargo build --timings -p sc-extract --profile dev-opt --features full`**
  — cheap diagnostic. Gives a per-crate + per-codegen-unit breakdown.
  Do this first.
- **Drop codegen-units to 16 or 64 temporarily** and re-measure.
  `codegen-units = 256` was picked for the pre-polymorphism type
  count and may be exactly wrong at 6k.
- **Disable serde derives on pool structs** to see if that's the bulk.
  Hacky — leave for last.

### `entities-scitem-ships` hub leaf: 34m

Single leaf feature, but the dev-opt cold build takes 34 minutes and
peaks at 10.5 GB. `ammoparams` at the same profile is 3m 58s / 2.4 GB.
Something about the ships closure is pathological — probably pulling
in most of `multi_feature` because ship records reference so much of
the DCB. This is the "hub leaf problem" flagged in
`docs/feature-gating-v2.md` §"Hub-leaf problem"; it's now measured
and worth documenting as a warning for anyone trying to depend on
this feature from a domain crate. Narrow consumers of ship data
should use more specific sub-features once we split this hub.

### Full build peak RAM at 25.7 GB

On a 32 GB machine this leaves ~6 GB headroom for the OS + browser +
VSCode. Any dev on 16 GB will OOM during `cargo build --features full`.
Worth noting in `CLAUDE.md` as a hard requirement. `cargo check` full
at 11.7 GB is much more manageable and still useful for iteration.

## Interpretation

- **Default cold check should be significantly faster than full cold check.**
  The v2 feature-gating design puts most types behind per-type
  `cfg(any(...))` unions, so with no features enabled rustc strips
  most of `multi_feature/` during parsing. If default approaches full,
  the classifier has drifted toward schema-static and wants
  investigation.
- **`dormant` should be marginally slower than `full`.** Adds 809
  types whose references are already in-scope via the `full` forward.
  If `dormant` is wildly slower than `full`, the dormant set grew
  unexpectedly — check the generator's dormancy classification.
- **Runtime parse under default should return 0 records.** Not a bug
  — no record extractor dispatch is compiled when no leaf feature is
  enabled.
- **Runtime parse under `--features full` is the realistic consumer
  number.** This is what a snapshot-building CI job or a domain crate
  like sc-weapons will see.
- **Snapshot size with graph ≫ without.** The reference graph
  (~1.4M edges) dominates the msgpack payload for the default config.
  With graph off, snapshots stay in the single-digit-MB range.
- **Peak RAM during `release` build of `full` is the canonical
  "how much memory does rustc need here" number.** If this ever
  exceeds ~12 GB, domain-crate authors on 16 GB machines will start
  hitting paging. Current target: stay under 10 GB.

#### `full` feature set — thin vs fat spot check

One-off comparison on `-Features full` to see whether fat LTO's cost
scales with IR volume the way the textbook predicts. `.cargo/config.toml`
carries `link-args=/STACK:8388608` for the Windows 8 MB runtime stack
(the 1 MB default overflows during `full` DCB extract dispatch).

| When | LTO | Check | Release build | Parse (std) | Parse (+graph) | Peak build RAM | Peak runtime RAM |
|---|---|---:|---:|---:|---:|---:|---:|
| 2026-04-15 23:42 | `thin` | 41.35s | 2m 5.5s | 32.84s | 38.10s | 5.03 GB | 4.64 GB |
| 2026-04-15 23:58 | `fat` | 40.16s | 2m 8.3s | 30.97s | 36.81s | 5.05 GB | 4.65 GB |

**Takeaway:** fat and thin are essentially tied on compile time even at
`full` (+2.2% for fat). Prediction that fat would be 2–3× slower than
thin on the larger workload was wrong — post-derive-drop IR volume is
small enough per type that fat LTO's single-threaded link pass doesn't
dominate. Fat is marginally faster at runtime (−6% parse std, −3%
parse+graph) and RAM is identical. **Fat LTO is the winner** on both
`entities-scitem-ships` and `full`; locking it in.

Follow-up: on `entities-scitem-ships`, opt=1 and opt=3 were
indistinguishable under fat LTO. Re-running on `full` to see if that
holds when IR volume is ~4× larger. The fat+opt=3 baseline is the
`fat` row from the block above.

| When | `sc-extract-generated.opt-level` | Check | Release build | Parse (std) | Parse (+graph) | Peak build RAM | Peak runtime RAM |
|---|---|---:|---:|---:|---:|---:|---:|
| 2026-04-15 23:58 | `3` (release default, no override) | 40.16s | 2m 8.3s | 30.97s | 36.81s | 5.05 GB | 4.65 GB |

**Takeaway:** opt=1 and opt=3 are indistinguishable on `full` too —
build time is literally tied (0.1s), runtime is within 3% (noise),

on a crate that feeds into the LTO pass doesn't materially affect
the output. Only opt=0 is a useful lever (it skips rustc passes
entirely and relies on fat LTO for everything, trading ~10% runtime
for ~34% compile time).
